# Lab 11: Memory и Context Management

## Цель

Реализовать две вещи без оверинжиниринга:

1. **Condense** — сжатие истории внутри одного запуска агента, когда подходит лимит окна модели (см. [гл. 13](../../book/13-context-engineering/README.md)).
2. **Long-term memory** — заметки агента, переживающие перезапуск процесса. Доступны как обычные инструменты (`memory.save` / `memory.recall`), а не как «слой контекста» (см. [гл. 12](../../book/12-agent-memory/README.md)).

Мы **не делаем**:

- ❌ `LayeredContext` (Working / Summary / Facts слои в один промпт).
- ❌ Автоматическое извлечение фактов LLM из каждого сообщения.
- ❌ Динамический system prompt с «текущим состоянием».
- ❌ Скоринг важности сообщений и переупорядочивание истории.

Почему — подробно в [гл. 13: Типовые ошибки](../../book/13-context-engineering/README.md#типовые-ошибки). Если коротко: всё перечисленное ломает prompt cache, ломает пары `tool_call ↔ tool_result` и почти всегда — попытка решить выдуманную проблему.

## Theory

### Два горизонта памяти

| Горизонт | Срок жизни | Кто пишет | Где живёт |
|---|---|---|---|
| **In-Run (краткосрочная)** | один запуск агента | Runtime автоматически | `[]Message` в памяти |
| **Long-term (долгосрочная)** | переживает перезапуск | агент явно вызывает tool | файл / БД / KV |

In-Run память — это **линейный иммутабельный массив сообщений** (system → user → assistant → tool → assistant → …). Когда `usage.PromptTokens` приближается к границе окна (например, 80%) или провайдер вернул `ContextOverflowError` — делаем **один `condense`**: берём середину истории, просим LLM суммировать, кладём результат как `user`-сообщение «контекст предыдущей работы», оставляем последние N сообщений целыми (важно — не разрывая `tool_call ↔ tool_result`).

Long-term память — это **не автомагия**. Это два tool'а, которые агент вызывает осознанно: «запомни X», «вспомни про Y». Так же, как новый сотрудник делает заметки в личном Notion. Если агент не вызвал tool — ничего не запомнили. И это правильно: нам не нужно копить мусор «пользователь сказал привет в 14:32».

### Почему condense, а не truncate

Простое отрезание `messages[-N:]` ломает пары tool-вызовов: вы можете обрезать ровно между `tool_call` от ассистента и `tool_result` от runtime — и провайдер вернёт 400. Кроме того, вы выкидываете контекст без следа.

Condense:

1. Сохраняет system prompt целиком.
2. Заменяет середину одним `user`-сообщением: «Контекст предыдущей работы: …».
3. Сохраняет последние N сообщений целыми, **проверяя** что не разорвали tool-пары.

## Задание

В `main.go` реализуйте простой агент со следующими возможностями.

### Часть 1: Linear memory + token tracking

Структура `Run` хранит только линейную историю одного запуска:

```go
type Run struct {
    messages    []openai.ChatCompletionMessage // линейный лог, system на индексе 0
    lastTokens  int                            // usage.PromptTokens последнего ответа
    contextMax  int                            // окно модели, например 128_000
    condenseDone bool                          // лимит: один condense на Run
}
```

Требования:

- `system` сообщение фиксируется в `messages[0]` и **не меняется** в течение Run.
- После каждого ответа провайдера обновляйте `lastTokens = resp.Usage.PromptTokens` — это первоисточник, а не свои оценки `len(content)/3`.
- Перед каждым следующим запросом проверяйте: `lastTokens > contextMax * 0.80` → запустить `condense` (один раз на Run).

### Часть 2: Реализация condense

```go
func (r *Run) condense(ctx context.Context, llm LLM) error {
    if r.condenseDone || len(r.messages) < 6 {
        return nil
    }
    system := r.messages[0]
    tail := r.messages[len(r.messages)-4:]    // последние 4 целиком
    head := r.messages[1 : len(r.messages)-4] // всё, что между

    // Гарантируем, что tail начинается с целой пары:
    // если первое сообщение в tail — tool result без своего tool_call в tail,
    // сдвиньте границу влево, чтобы пара осталась цела.
    tail = ensureToolPairs(r.messages, tail)
    head = r.messages[1 : len(r.messages)-len(tail)]

    summary, err := summarizeWithLLM(ctx, llm, head)
    if err != nil {
        return err
    }

    next := make([]openai.ChatCompletionMessage, 0, 2+len(tail))
    next = append(next, system)
    next = append(next, openai.ChatCompletionMessage{
        Role:    "user",
        Content: "Контекст предыдущей работы:\n\n" + summary,
    })
    next = append(next, tail...)

    r.messages = next
    r.condenseDone = true
    return nil
}
```

Промпт для `summarizeWithLLM`:

```text
Сожми переписку в краткую справку для следующего шага агента.
Сохрани:
1. Что было поручено (исходная задача).
2. Какие решения уже приняты и почему.
3. Какие файлы / ресурсы уже прочитаны и что в них релевантно.
4. Что ещё осталось сделать.

Не пересказывай вежливости, опусти служебный шум.
```

Реактивная ветка: если провайдер вернул `ContextOverflowError`, выполните `condense` и **повторите** запрос один раз. Если снова overflow — fail с понятной ошибкой (агенту дали слишком много за один шаг, дробите задачу).

### Часть 3: Long-term memory как инструмент

Заведите хранилище — JSON-файл или SQLite. Минимальный API:

```go
type Store interface {
    Save(ctx context.Context, key, value string) error
    Recall(ctx context.Context, query string) ([]Entry, error)
    Delete(ctx context.Context, key string) error
}

type Entry struct {
    Key       string
    Value     string
    CreatedAt time.Time
}
```

Регистрация инструментов агенту:

- **`memory.save`** — параметры `key` (string), `value` (string). Возвращает «ok» или ошибку.
- **`memory.recall`** — параметр `query` (string). Возвращает до 5 записей. Поиск может быть подстрокой по `key+value` — для лабы достаточно, embeddings избыточны.
- **`memory.delete`** — параметр `key`. Полезно для самокоррекции.

Никакого автозаписывания фактов LLM-ом. Агент сам решает, что класть в память, через `memory.save`. Это и проще, и честнее — вы видите в логах, что и зачем агент решил запомнить.

В system prompt дайте агенту **короткую** инструкцию (1-2 предложения):

```text
У тебя есть инструменты memory.save / memory.recall / memory.delete для заметок,
которые переживают перезапуск. Используй их для устойчивых фактов о пользователе
и проекте. Не сохраняй сиюминутные статусы.
```

Не вшивайте текущее содержимое памяти в system prompt — иначе при каждой записи у вас инвалидируется prompt cache. Память подгружается через `memory.recall`, когда агент сам её спросит.

### Часть 4: Минимальный CLI

`go run ./main.go --memory ./memory.json`

REPL: пользователь вводит сообщения, агент отвечает, может вызвать `memory.*` и обычные tool'ы. После выхода и повторного запуска `memory.recall` должен находить старые записи.

## Что проверить руками

1. Запустите длинный диалог так, чтобы `usage.PromptTokens` перевалил за 80% — убедитесь, что `condense` сработал ровно один раз и `system[0]` остался байт-в-байт прежним.
2. Сэмулируйте `ContextOverflowError` (например, искусственно занизьте `contextMax`) — убедитесь, что реактивный condense запускается и запрос повторяется.
3. Перезапустите процесс. Спросите агента про факт, который сохраняли в прошлой сессии — он должен сам вызвать `memory.recall` и ответить.
4. Проверьте, что `system` prompt не содержит ни текущего времени, ни «текущего состояния», ни содержимого памяти. Только роль, правила, описание инструментов.

## Типовые ошибки в этой лабе

- **Считать токены через `len(content)/3`.** Используйте `resp.Usage.PromptTokens` — это то, что реально засчитал провайдер. Свой счётчик годится только для прикидки до отправки.
- **Делать `messages = messages[len(messages)-N:]`.** Сломаете пары tool-вызовов и получите 400. Используйте `condense` или аккуратное окно с проверкой пар.
- **Класть содержимое памяти в system prompt.** Каждая запись инвалидирует prompt cache — стоимость и латентность улетают вверх.
- **Делать второй, третий, четвёртый condense подряд.** Если после первого condense вы снова упёрлись в окно — проблема не в памяти, а в том, что задача не дробится. Откажите явно.
- **Извлекать «факты» LLM-ом из каждого сообщения автоматически.** Получите шум вроде «пользователь сказал спасибо». Сохранение — осознанное действие агента через tool.

## Критерии сдачи

**Сдано:**

- [x] `system` сообщение в `messages[0]` стабильно весь Run (байт-в-байт).
- [x] Решение «пора сжимать» принимается по `usage.PromptTokens`, не по своим оценкам.
- [x] Условие срабатывания: `lastTokens > contextMax * 0.80` **или** реактивно на `ContextOverflowError`.
- [x] `condense` выполняется не более одного раза на Run.
- [x] После `condense` сохраняются: system, summary как user-сообщение, последние N сообщений; пары `tool_call ↔ tool_result` не разорваны.
- [x] Long-term memory доступна агенту как 3 инструмента (`save` / `recall` / `delete`), не как часть system prompt.
- [x] Память переживает перезапуск процесса.

**Не сдано:**

- [ ] `LayeredContext` или любая динамическая склейка system prompt.
- [ ] Авто-извлечение фактов из каждого сообщения.
- [ ] Truncate без проверки tool-пар.
- [ ] Содержимое памяти склеивается в system prompt при каждом запросе.
- [ ] Несколько condense подряд в одном Run без явного лимита.
- [ ] Подсчёт окна через `len(content)/3` вместо `usage.PromptTokens`.
